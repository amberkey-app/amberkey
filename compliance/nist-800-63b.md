# NIST SP 800-63B alignment (digital identity / authentication)

Last reviewed: 2026-07-11 · Self-assessed; no third-party audit.

AmberKey's account authentication is **aligned with SP 800-63B AAL2
authenticator requirements**, with one deliberate, documented deviation on
session length. Each claim below describes the running service's behavior and is
observable in the app's authentication flow.

## Where we align

| 800-63B topic | AmberKey behavior |
|---|---|
| Authenticator type | **Passkey-first (WebAuthn)**: a multi-factor cryptographic authenticator with user verification — the strongest authenticator class 800-63B describes, and **phishing-resistant** (verifier-impersonation-resistant). Account creation is passkey-only. |
| Fallback | **TOTP** (RFC 6238, 30s step, ±1 step skew) as a sign-in fallback only, enrollable after account creation. Codes are **single-use** (replay of the same time step is rejected) and **rate-limited** per account. |
| SMS for authentication | **None.** SMS is used for liveness notifications only, never as an authentication factor — avoiding the channel 800-63B restricts. |
| Passwords | **None exist**, so 800-63B's memorized-secret rules (no composition rules, no forced rotation, blocklists) are satisfied vacuously — the failure modes are absent by construction. |
| Secret storage | The server stores WebAuthn **public keys** and hashed session tokens (SHA-256); TOTP secrets are stored server-side as required by the algorithm. No password hashes exist. |
| Reauthentication | **Step-up re-authentication** (fresh passkey or TOTP within 10 minutes) is required for sensitive state changes: recovery-circle edits, liveness-schedule changes, duress-code changes, contact changes. This follows 800-63B's reauthentication-before-sensitive-operations guidance. |
| Verifier compromise resistance | Passkey private keys never leave the user's authenticator; a full server compromise yields no reusable authentication secrets for passkey users. |

## Deliberate deviation

- **Session length.** 800-63B suggests AAL2 sessions reauthenticate at ~12
  hours or 30 minutes of inactivity. AmberKey uses a **30-day rolling session**.
  This is a considered product decision, not an oversight: AmberKey is a
  consumer dead-man's-switch, and aggressive logout increases missed check-ins —
  the product's most dangerous failure mode (a false death escalation). The
  compensating controls are the step-up requirement on every sensitive action
  (above) and cookie hardening (HttpOnly, Secure, SameSite=Strict). We state
  this deviation rather than hide it.

## Scope

This assessment covers the owner console's authentication. The shareholder
portal is intentionally sessionless (rate-limited single-purpose links), and the
offline recovery tool has no authentication at all — by design, since it must
work when AmberKey doesn't exist.

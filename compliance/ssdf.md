# NIST SP 800-218 SSDF alignment (secure software development)

Last reviewed: 2026-07-11 ¬∑ Self-assessed; the evidence is this repo's CI.

Mapping of AmberKey's development practice to the SSDF's four practice groups.
Every claim points at a file in this repository or a CI job that runs on every
push (`.github/workflows/ci.yml`, executed by our own runner).

## PO ‚Äî Prepare the Organization

- **Security requirements are written down first:** the threat model and the
  blind-server invariant are specification documents (`spec/`, the public
  security page), not retrofits.
- **Disclosure channel:** `SECURITY.md` + `security@amberkey.app`.
- **Toolchain policy:** Rust stable, pinned dependencies via lockfiles
  (`Cargo.lock`, `package-lock.json`) committed and used in CI and release builds.

## PS ‚Äî Protect the Software

- **Source integrity:** all changes flow through pull requests into a
  protected `main` (required green CI, no direct pushes, no force-push); release
  history is public for the open subset.
- **Release integrity:** versioned tags ‚Üí CI-built container images pinned by
  tag in production (`ops/RELEASING.md`); the offline recovery tool is built
  **reproducibly** ‚Äî CI builds it twice and fails on hash mismatch ‚Äî and every
  release is **minisign-signed**, with the signing key fingerprint printed on
  physical share cards so survivors can verify the tool without trusting us.
- **Archival:** the open subset is mirrored (GitHub, Codeberg, Software
  Heritage, Internet Archive) so artifacts outlive any single host.

## PW ‚Äî Produce Well-Secured Software

- **Static analysis on every push:** `cargo clippy --workspace --all-targets
  -- -D warnings` (warnings are build failures).
- **Tests as gates, including reference vectors:** the full official Trezor
  SLIP-39 vector suite, age round-trip tests, server integration tests
  (auth/step-up, invite gate, duress, ceremony), and end-to-end recovery drills
  (including a fully offline `file://` recovery with networking disabled).
- **Memory safety:** Rust for all cryptographic and server code; the only
  unsafe-adjacent surface is the WASM boundary generated by wasm-bindgen.
- **Least dependency:** the client's only crypto path is the Rust core compiled
  to WASM ‚Äî no JS crypto; the marketing site ships zero third-party scripts.

## RV ‚Äî Respond to Vulnerabilities

- **Dependency scanning on every push:** `cargo audit` and
  `npm audit --audit-level=high` fail the build on known-vulnerable deps.
- **Coordinated disclosure:** documented process in `SECURITY.md`.
- **Patch path:** pinned-image releases mean a fix is one tag away from
  production, with instant rollback to the previous tag.

## Known gaps (tracked, not hidden)

- No SBOM published yet; no SLSA provenance attestations yet (reproducible
  builds + signatures today; provenance is on the roadmap ‚Äî see the trust page).
- Dependency updates are manual (no bot cadence yet).
- These and the broader operational gaps live in `ops/SOC2-PHASE0.md`.

# Build provenance (SLSA-informed, self-hosted builder)

Last reviewed: 2026-07-11 · Self-assessed. Precise claim: **signed SLSA v1
provenance is generated by CI for every release** — consistent with the SLSA
Build L2 requirements, with the one structural caveat stated below. We do not
claim a SLSA level outright.

## What is produced

On every release tag (`v*`), the image workflow generates an
**in-toto Statement** (`https://slsa.dev/provenance/v1` predicate) naming:

- the pushed `amberkey-server` and `amberkey-caddy` **image digests**,
- the **`recover.html`** inside that release (its SHA-256 — the same value
  printed on physical share cards),
- the exact **source commit** and tag, the workflow path, and the run id.

The statement is signed with a dedicated **CI provenance key** (SSHSIG — plain
OpenSSH, so anyone can verify with stock tooling) and uploaded to the package
registry at `amberkey-provenance/<tag>/provenance-<tag>.json[.sig]`.

This key signs *provenance only*. Release artifacts themselves are signed by
the separate operator-held **minisign release key** (fingerprint printed on
cards); compromising CI does not forge releases, and compromising the release
key does not forge provenance.

## Verifying

```sh
# public key (also in this repo's history):
# ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGcdjCnZXwt6TETPBUHlavo6O2qQ2EEGoxUylKn+LFYJ amberkey-ci-provenance
printf 'ci@amberkey.app ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGcdjCnZXwt6TETPBUHlavo6O2qQ2EEGoxUylKn+LFYJ\n' > allowed_signers
ssh-keygen -Y verify -f allowed_signers -I ci@amberkey.app -n amberkey-provenance \
  -s provenance-vX.Y.Z.json.sig < provenance-vX.Y.Z.json
# then check the digests in "subject" against what you're running:
docker inspect --format '{{index .RepoDigests 0}}' git.dx.pe/tarnover/amberkey-server:vX.Y.Z
sha256sum recover.html
```

## Honest caveats

- **Self-hosted builder.** SLSA Build L2 expects a *hosted* build platform;
  ours is our own Forgejo Actions runner. The provenance is real and signed,
  but "the platform attests" and "we attest" are closer together than in a
  hosted setup — which is why we say *SLSA-informed* rather than "SLSA L2".
- The provenance signing key lives as a CI secret; rotation is documented in
  the ops runbooks.
- Complementary and pre-existing: the recovery tool is **reproducible** (CI
  builds twice, compares hashes), so anyone can independently regenerate the
  `recover.html` digest from the tagged source — a stronger check than any
  attestation.
